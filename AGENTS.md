# AGENTS.md

## 项目定位

本项目是“有个饭局”：面向高端商务饭局的助理、礼宾、氛围协同与流程管理产品。它不是餐厅订座、陪玩、夜场、陪侍或暧昧交易平台。所有产品、文案、界面和代码命名都必须服务于“体面、克制、商务、可信、合规”的定位。

## 业务边界

- 可以表达：商务接待、饭局筹备、礼宾接待、座次建议、话题提醒、流程协同、饭后复盘、人情世故、关系经营、边界感。
- 禁止表达：陪酒、陪侍、陪玩、暧昧交易、色情暗示、夜场服务、行贿送礼、灰色资源、保证办成事、权钱交易。
- 不把用户需求描述成“购买某个人”。应描述为“配置服务角色、礼宾流程、现场协同、氛围维护”。
- 不制造阶层羞辱、性别对立、PUA、违法违规或厚黑学表达。
- 涉及隐私、身份、客户资料、支付、订单、评价、聊天和联系方式时，默认按高敏感数据处理。

## 产品原则

- 优先让用户感到“这事有人稳妥处理”，而不是“这平台很热闹”。
- 功能设计要减少尴尬、降低不确定性、提高体面度。
- 用户侧语言要克制、明确、低噪音；避免油腻、江湖气和过度营销。
- 小程序审核风险优先级高于短期转化。任何可能被理解为陪侍、交易人身服务或灰产的表达都要改写。

## 技术结构

- `apps/mp-client`: 微信小程序客户端。
- `apps/api`: 后端 API，NestJS/TypeScript。
- `apps/admin-web`: 管理后台，Vue/Vite。
- `brand`: 品牌设计、Logo 和设计自动化资料。
- `YinQiTong`: 可能是同步或镜像仓库，修改前先确认任务是否明确指向该目录。

## 开发规则

- 修改前先阅读相关页面、组件、接口和现有命名，不凭空新建风格。
- 保持改动范围小，只改当前任务需要的文件。
- 不要重置、覆盖或删除用户已有改动。当前工作区可能长期存在未提交修改。
- 不要自动提交、推送或发布，除非用户明确要求。
- 对订单、支付、用户资料、权限、审核相关改动要说明风险点并补充校验。
- UI 文案应使用“商务饭局、礼宾、助理、协同、控场、复盘、边界”等词，避免低俗化表达。

## 常用检查命令

在相关目录分别运行可用命令，不要假设根目录有统一脚本：

```bash
cd apps/api && npm run typecheck && npm test
cd apps/mp-client && npm run typecheck
cd apps/admin-web && npm run typecheck && npm run build
```

如果依赖缺失或命令失败，先报告具体失败原因；不要静默跳过。

## 输出要求

完成任务后说明：

- 改了哪些文件。
- 用户可见行为有什么变化。
- 跑了哪些检查命令，结果如何。
- 如果没能跑检查，说明原因。
- 是否存在微信审核、隐私、支付或合规风险。
